- 개요 → 핵심 개념 → 실무 포인트 → TODO 순으로 서술하고, 명령어·경로는 코드 서식으로 표시합니다.

## 기본 메타데이터
모든 노트 상단에 다음 정보를 명시합니다.
```
- 마지막 업데이트: YYYY-MM-DD
- 상태: 초안/검토중/완료 중 택1 (필요시 부가 설명)
```

## 파일명 규칙
- 주제를 명확히 드러내는 영어 혹은 숫자 결합 사용 (`process_scheduling.md`, `tls_handshake.md` 등).
- 띄어쓰기 대신 하이픈(`-`) 혹은 언더스코어(`_`) 사용.
- 동일 주제의 변형 문서가 생기지 않도록 기존 파일 검색 후 작성.

## 문서 구조 권장안
1. **개요**: 주제 요약, 왜 중요한지 2~3문장.
2. **핵심 개념 / 절차**: 목록 또는 표 형태로 정리.
3. **실무/시험 포인트**: 기억해야 할 값, 공식, 주의 사항.
4. **TODO/후속 연구**: 미완료 과제, 참고 링크.
5. 필요한 경우 다이어그램이나 코드 블록을 추가하되, 경로와 작성 도구를 기록.

### 다이어그램 관리 가이드
- draw.io 파일은 `drawio/<주제>.drawio` 경로에 저장하고, 문서에는 파일 경로와 마지막 업데이트 날짜를 명시합니다.
- 다이어그램을 공유할 때는 `File > Export as > PNG`로 내보내 `docs/assets/<주제>.png`에 보관한 뒤 `![설명](../docs/assets/<주제>.png)` 형태로 삽입하세요.
- 다이어그램이 문서의 주요 흐름을 설명한다면 TL;DR 또는 "예제/다이어그램" 섹션에서 바로 확인할 수 있도록 링크를 제공합니다.
- 버전이 바뀌었을 때는 draw.io 파일과 PNG 모두를 갱신하고, 문서의 "마지막 업데이트" 날짜를 함께 수정합니다.
- draw.io를 사용할 수 없는 환경에서는 동일 내용을 Mermaid 다이어그램으로 제공하고, 추후 PNG 갱신 시 Mermaid 블록과 export 이미지가 동일한 내용을 담도록 유지합니다.

## 서식 & 요약 가이드
- 개요 아래에 한 문장 TL;DR 요약을 추가해 핵심 결론을 빠르게 전달합니다.
- 명령어, 파일 경로, 구성 값은 인라인 백틱(`) 또는 코드 블록으로 감싸 가독성을 높입니다.
- 표가 길어지는 경우 가장 중요한 행부터 배치하고, 불필요한 중복 텍스트는 각주나 링크로 치환합니다.
- 외부 자료를 반복 참조할 때는 동일한 링크명을 재사용해 사용자가 문맥을 쉽게 추적하도록 합니다.

### TL;DR & 코드 서식 예시
```
- 마지막 업데이트: 2025-09-24
- 상태: 검토중

## TL;DR
- 주요 포트는 `iptables`에서 `ACCEPT` 정책으로 열어 두고, 나머지는 `DROP`으로 기본 설정합니다.
```
- 예시처럼 TL;DR은 1~2줄이면 충분하며, 명령어·경로(`iptables`, `/etc/hosts`)는 백틱으로 감싸 주세요.

## 링크 및 참고 자료
- 외부 링크는 마크다운 하이퍼링크 사용, 간단한 설명 추가.

## 버전 관리 메모
- 큰 개정 후에는 `README.md`의 “진행 중 과제” 섹션 갱신.
- 대규모 삭제/이동 전에는 기존 내용을 `archive/` 폴더에 보존하거나 Git 태그를 고려.

## 연계 문서 안내
- `docs/getting_started.md`: 신규 사용자가 읽는 첫 문서입니다. 주요 탐색 경로가 바뀌면 단계별 안내와 표를 함께 갱신하세요.
- `docs/document_inventory.md`: 폴더 현황, 미완성 문서, 분기 점검 체크리스트를 이 문서에서 추적합니다.
- `docs/note_template.md`: 새 노트 작성 시 복사해 사용하는 기본 템플릿입니다. 섹션 구조 변경이 필요하면 템플릿부터 업데이트하세요.
- `README.md`: “구조 한눈에 보기” 표와 “빠르게 시작하기” 섹션이 모든 폴더 안내의 기준이 되므로 변경 시 우선 반영합니다.
- `security_note/README.md` 및 하위 `attack/`, `defend/` README: 보안 전술/대응 문서의 대표 링크 표를 유지합니다.
